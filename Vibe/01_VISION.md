# AI 知识库系统 — 项目愿景文档

## 1. 项目背景

证券行业基础设施文档具有**多源、异构、高频更新**的特点：

| 数据源 | 核心内容 | 更新特征 |
|--------|----------|----------|
| 上交所-交易技术支持专区 | 技术通知、接口规范、软件下载、测试文档、技术杂志 | 月度/不定期发布 PDF，历史版本需归档 |
| 深交所-技术服务 | 技术公告、数据接口、服务指引、第五代交易系统文档、金融科技专栏 | 跨年持续更新，版本号管理 |
| 中国结算-业务规则 | 账户管理、登记存管、清算交收、债券、期权等 10+ 类规章 | 法规频繁修订，需追踪废止状态 |

**当前痛点：**
- 三个站点独立维护，从业者需手动跨站检索
- PDF / Word / ZIP 等异构格式无法统一全文搜索
- 版本迭代缺乏自动追踪，错版风险高
- 技术通知与业务规则之间存在隐性关联（如接口规范的更新往往伴随结算规则调整），人工难以发现

---

## 2. 愿景陈述

> **构建一套 Agent 协作的智能知识库系统，自动爬取上交所、深交所、中国结算三大官方来源的技术与规则文档，完成 采集→分析→整理→发布 全链路自动化，使从业者在一个入口即可精确检索、版本追溯、关联发现最新最全的行业知识。**

### 核心价值

- **全** — 不遗漏任何一个技术通知、接口版本或规则修订
- **准** — 版本差异自动比对，废止规则自动标记
- **快** — 从发布到入库 < 30 分钟，告别手动下载整理
- **智** — Agent 自动发现跨站跨文档的内容关联（如接口修订→结算规则调整→技术指南更新）

---

## 3. 范围

### 3.1 纳入范围

| 维度 | 内容 |
|------|------|
| 数据源 | 上交所交易技术支持专区（通知/接口/指南/软件/测试/杂志） |
| | 深交所技术服务（公告/接口/指引/指南/金融科技） |
| | 中国结算业务规则（全部 12 个子分类，含已废止规则） |
| 文件格式 | HTML 页面、PDF、Word（.doc/.docx）、ZIP 压缩包 |
| 采集频率 | 通知/公告类：每日轮询；接口/规则文档：变更即触发更新 |
| 知识加工 | 全文索引、版本追溯、废止标记、跨站关联 |
| 发布形态 | Web 搜索门户 + REST API + Webhook 增量推送 |

### 3.2 不纳入范围（首期）

- 非上述三个站点的其他监管机构文档（如证监会、中金所等）
- 投资人教育、培训视频、市场数据类内容
- 用户登录/权限管理（首期为公开展示，后续可扩展）

### Agent 职责说明

| Agent | 职责 | 关键技术 |
|-------|------|----------|
| **Orchestrator** | 编排整个工作流，判断增量/全量，错误重试与告警 | DAG 任务调度，状态机 |
| **Crawler** | 定向爬取三个站点，识别页面更新，下载附件 | Python Scrapy / Playwright，增量去重 |
| **Parser** | 将 PDF/Word/HTML 统一提取为结构化 Markdown + 元数据 | PyMuPDF / python-docx / Pandoc，LLM 辅助提取 |
| **Analyzer** | 版本差异分析、废止检测、跨站实体关联 | diff 引擎 + 向量语义匹配 + LLM 分类 |
| **Storage** | 版本化管理文档，向量化存储，全文索引 | PostgreSQL + pgvector + Elasticsearch |
| **Publisher** | 提供搜索门户与 API，推送更新通知 | FastAPI + Vue.js + WebSocket |

## 5. 数据模型（核心实体）

```
Document
├── id: UUID
├── source: enum(sse|szse|chinaclear)
├── source_url: string
├── title: string
├── category: string        (如 "技术通知", "数据接口", "清算与交收")
├── publish_date: datetime
├── fetch_date: datetime
├── format: enum(html|pdf|doc|docx|zip)
├── version: string
├── is_deprecated: boolean
├── superseded_by: UUID?    (被哪个版本替代)
├── raw_path: string        (原始文件存储路径)
├── parsed_content: text    (Markdown 化全文)
└── embedding: vector(1536) (语义向量)

DocumentRelation
├── source_id: UUID
├── target_id: UUID
├── relation_type: enum(supersedes|references|contradicts|same_topic)
└── confidence: float
```

## 6. 路标规划

| 阶段 | 目标 | 时间 |
|------|------|------|
| **P0 — MVP** | 打通 Crawler → Parser → Search，三个站点各覆盖 50+ 核心文档，实现全文搜索 | 6 周 |
| **P1 — 增量** | 定时增量采集+变更检测，Analyzer Agent 上线，支持版本追溯和废止标记 | 4 周 |
| **P2 — 关联** | 跨站实体关联引擎上线，Relation 模型投入使用，搜索支持"相关内容"推荐 | 4 周 |
| **P3 — 发布** | Web 门户 + API + Webhook 发布，支持个人/团队订阅特定分类更新通知 | 4 周 |
| **P4 — 持续** | 覆盖 95%+ 历史文档，定期模型微调，引入用户反馈闭环 | 持续 |

## 7. 成功指标

| 指标 | 目标 |
|------|------|
| 数据源覆盖率 | ≥ 95%（按三个站点文档总数计） |

| 跨站推荐准确率 | ≥ 80%（A/B 测试） |
| 用户搜索满意度 | NPS ≥ 40 |

## 8. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 目标网站改版导致爬虫失效 | 爬虫层配置化，Parser 独立，Orchestrator 监控爬取成功率，异常自动告警 |
| PDF/扫描件 OCR 质量参差 | Parser Agent 先尝试直接提取，降级时调用 OCR（PaddleOCR/Tesseract），LLM 后处理校正 |
| 法规更新频繁，废止判断复杂 | Analyzer Agent 双向验证：① 页面显式废止标注 ② 语义匹配新替代文档，人工审核兜底 |
| 合规与速率限制 | Crawler 遵循 robots.txt，控制请求频率，使用合法 UA 标识 |
