# 评分标准全面评估报告

基于 `knowledge/articles/sse/entries/` 下 **171 个实际条目**的数据驱动分析，逐维度评估当前评分标准的合理性。

---

## 总体评估

| 维度 | 当前满分 | 评估结论 | 建议 |
|------|----------|----------|------|
| 摘要质量 | 25 | ⚠️ 区分度低，阈值不合理 | **调整评分规则** |
| 技术深度 | 25 | ❌ 核心逻辑与业务目标背离 | **替换为「变更完整性」** |
| 格式规范 | 20 | ✅ 基本合理 | **微调子项权重** |
| 标签精度 | 15 | ❌ 核心假设与数据矛盾 | **替换为「可检索性」** |
| 空洞词检测 | 15 | ❌ 黑名单不适合领域文档 | **替换为「信息密度」** |

---

## 维度一：摘要质量（25 分）

### 数据事实

| 指标 | 数值 |
|------|------|
| 平均长度 | **36 字** |
| < 20 字 | **0 条**（0%） |
| 20~49 字 | **151 条（88%）** |
| >= 50 字 | **20 条（12%）** |
| 含"初始版本" | **78 条（46%）** |

### 问题分析

1. **评分规则无区分度** — 88% 的条目都落在 20~49 字区间，全部拿 10 分，拉不开差距。实际区分只在 +5（技术关键词奖励）环节产生。
2. **"初始版本" 不应得 0 分** — 在知识条目的语境下，"初始版本" 是一个 **有效元数据**，表示该文档是首次发布、无历史版本可追溯。得 0 分不合理。
3. **技术关键词奖励太泛** — "接口""协议"几乎是所有技术文档都会出现的词，几乎没有区分作用。

### 建议

**保留维度但重构评分逻辑：**

| 条件 | 得分 | 理由 |
|------|------|------|
| summary 为空 | 0 | 确实缺数据 |
| 仅含"初始版本"等 | **12** | 有效初始状态，非缺陷 |
| 正常摘要（无变更关键词） | 15 | 信息完整但不含变更详情 |
| 含具体变更描述 | +5~10 | 包含接口编号 `IS\d+` 或版本号 `v\d+` 或字段名变更 |
| 上限 | 25 | — |

## 维度二：技术深度（25 分）

### 数据事实

| 指标 | 数值 |
|------|------|
| 平均 content_md 长度 | **43,886 字** |
| >= 5,000 字 | **100 条（58%）** |
| 2,000~4,999 字 | **12 条（7%）** |
| 500~1,999 字 | **59 条（35%）** |
| < 500 字 | **0 条** |
| **有 `score` 字段的条目** | **0 条** |

### 问题分析

1. **核心逻辑错误** — 当前用 content_markdown **字符长度** 衡量"技术深度"完全不成立。一份 10,000 字的公告模板会比一份 2,000 字的精确接口规范得分更高，与实际业务价值相反。
2. **完全无区分度** — 100 条 >=5,000 字全部拿 18 基础分 + 密度奖励。技术术语密度奖励中"接口""协议"等词几乎出现在所有文档中，奖励变成保底分。
3. **与核心业务目标不符** — 知识库的核心价值是**变更追踪**，不是文档长短。"技术深度"命题本身不适合证券文档分类场景——业务规则文档可能短但重要，接口规范可能长但条目化。

### 建议

**替换为「变更完整性」(Change Completeness, 25 分)：**

| 子项 | 满分 | 评估逻辑 |
|------|------|----------|
| 版本链完整 | 8 | `previous_version` 或 `superseded_by` 正确填写 |
| 变更状态一致 | 7 | status 与对应字段一致（deprecated→有 deprecated_date） |
| 变更内容实际 | 5 | content_markdown 不是纯公告模板（检测表格/列表/技术关键词密度） |
| has_changes 准确 | 5 | 有变更的条目正确标注了 `has_changes` 标签 |

这样评分直接衡量**知识库核心价值**的完成度。

## 维度三：格式规范（20 分）

### 数据事实

| 指标 | 数值 |
|------|------|
| 有 id 且格式正确 | 全部 |
| 有 title 且非空 | 全部 |
| 有 source_url | 全部 |
| status 在枚举内 | 全部 |
| public_date 存在 | 部分（含 00000000 占位） |
| crawl_date 存在 | 部分 |
| **superseded → 有 superseded_by** | **61/61（100%）** |
| **deprecated → 有 deprecated_date** | **0/26（0%）** |

### 问题分析

1. 五个子项整体设计合理，但 **source_url** 在这个项目中所有条目都有，没有区分度。
2. **deprecated_date 缺失**（26/26）是明确的模式缺陷，当前评分未覆盖。
3. **条件一致性检查**应该在"格式规范"维度中处理，而不是留给 validate_json。

### 建议

**微调子项分配：**

| 子项 | 满分 | 评估内容 |
|------|------|----------|
| id 格式 | 4 | 正则匹配（含 type 缩写一致性） |
| title 非空 | 4 | 已有逻辑 |
| source_url | 3 | 已有（降 1 分，因区分度低） |
| status + 版本链 | 5 | status 合法 + superseded→superseded_by + deprecated→deprecated_date |
| 时间戳 | 4 | public_date 存在（3分）+ crawl_date 存在（1分）|

## 维度四：标签精度（15 分）

### 数据事实

| 指标 | 数值 |
|------|------|
| 平均标签数 | **8.5 个** |
| 0 个标签 | **0 条** |
| 1~3 个标签 | **26 条（15%）** |
| >= 6 个标签 | **122 条（71%）** |
| 含 IS 编号标签 | **155 条（91%）** |
| 含业务领域标签 | **92 条（54%）** |
| title 中含 IS 编号 | **81 条（47%）** |

### 问题分析

1. **"1-3 个标签最佳" 完全错误** — 71% 的条目 >= 6 个标签，平均 8.5 个。在这个领域，一份接口文档需要同时标注：数据源（sse）+ 文档类型（guide）+ 接口编号（IS101, IS124）+ 业务领域（债券）+ 状态（has_changes）+ 系统（STEP），6~8 个标签是合理且必要的。
2. **"非法标签"惩罚误杀率高** — 当前白名单只有 60+ 个标签，而实际标签种类远超此数。一个条目如果有合法的新标签（如未来新的 IS 编号），会被错误扣分。
3. **标签的唯一价值是"可检索性"**——不在于数量多少，而在于覆盖面和特异性。

### 建议

**替换为「可检索性」(Retrievability, 15 分)：**

| 子项 | 满分 | 评估逻辑 |
|------|------|----------|
| 覆盖数据源标签 | 3 | 是否有 `{sse,szse,chinaclear}` 之一 |
| 覆盖类型标签 | 3 | 标签中是否包含文档类型（type 对应标签） |
| 接口/系统标签 | 3 | 是否有 `IS\d+` 或 `STEP/BINARY` 等特异性标签 |
| 业务领域标签 | 3 | 是否有业务类标签（债券/股权/期权等） |
| title 含关键标识 | 3 | title 是否包含接口编号、版本号等重要搜索入口 |

这样评分衡量的是：**这条记录能否被用户精确检索到**。

## 维度五：空洞词检测（15 分）

### 数据事实

| 指标 | 数值 |
|------|------|
| 命中总次数 | **59 次** |
| 涉及文件数 | **~30 个** |
| 命中词分布 | 几乎全是"对齐"和"强大的" |
| 营销黑话命中（赋能/抓手/闭环/底层逻辑） | **1 次** |
| 技术文档模板套话（未纳入检测） | **大量存在** |

### 问题分析

1. **黑名单错位** — "赋能""抓手""闭环"等互联网/管理咨询黑话在交易所技术文档中几乎不存在（仅"赋能"命中 1 次）。保留这些词只是增加代码复杂度，无实际作用。
2. **英文空洞词完全无效** — 交易所文档几乎没有英文抒情内容，"groundbreaking""game-changing" 等永远不会出现。
3. **误判"对齐"** — "对齐"在接口文档中是正常技术术语（如"数据对齐""字节对齐"），非空洞词。
4. **真正的领域空洞词未被覆盖**：
   - 公告模板套话：`特此通知`、`请各市场参与人遵照执行`、`有关事项通知如下`
   - 冗余开头：`为进一步加强...`、`根据中国证监会...`、`为...发布...通知如下`
   - 低信息密度段落：大段的"根据...相关规定..."而没有具体技术内容

### 建议

**替换为「信息密度」(Information Density, 15 分)：**

检测真正影响知识条目标质量的问题：

| 子项 | 满分 | 评估逻辑 |
|------|------|----------|
| 无公告模板套话 | 5 | 检测 `特此通知`、`请遵照执行`、`附件：` 等公文模板 |
| title/summary 不冗余 | 5 | title 信息与 summary 不高度重复 |
| 正文信息密度 | 5 | content_markdown 中有效技术内容（表格、代码、列表）占比 |

## 建议的 5 维度重构方案

| # | 维度 | 满分 | 核心问题 | 评估方法 |
|---|------|------|----------|----------|
| 1 | **摘要变更精度** | 25 | summary 是否捕获了变更实质 | 初始版本=12分 + 接口号/版本号/字段名奖励 |
| 2 | **变更完整性** | 25 | 版本链 + 状态一致性 + 变更内容实质 | 4 个子项（8+7+5+5） |
| 3 | **元数据规范** | 20 | 字段格式 + 条件一致性 | 5 个子项（4+4+3+5+4） |
| 4 | **可检索性** | 15 | 标签覆盖面 + title 标识性 | 5 个子项各 3 分 |
| 5 | **信息密度** | 15 | 无模板套话 + 无冗余 + 有效内容占比 | 3 个子项各 5 分 |

**主要改变：**
- 维度二从"技术深度"改为"变更完整性"——直接对齐核心业务价值
- 维度四从"标签精度"改为"可检索性"——聚焦搜索发现能力
- 维度五从"空洞词检测"改为"信息密度"——使用领域相关的内容质量指标
- 维度一调整评分阈值，适配"初始版本"占 46% 的实际分布

---

**说明：** 以上评估基于 SSE 171 条实际数据，如需覆盖 SZSE/ChinaClear 的数据，维度四的标准标签列表和维度五的模板套话黑名单可能需要按数据源补充调整。
